## 1. 게임 개요

스플렌더는 르네상스 시대 보석상이 되어 보석 토큰을 모으고, 그것으로 개발 카드(광산·운송·상점)를 구매해 **명성 점수(prestige)**를 쌓는 엔진 빌딩 게임이다. 카드를 살수록 영구 보너스(할인)가 쌓여 더 비싼 카드를 싸게 살 수 있고, 보너스가 일정 조건을 만족하면 귀족이 방문해 추가 점수를 준다. **먼저 15점에 도달**하면 그 라운드까지 마친 뒤 최고 점수자가 승리한다.

- 인원: 2~4인 / 플레이타임: 약 30분 / 연령: 10+

## 2. 게임 규칙 명세 (Specification) — **구현의 단일 진실 공급원**

> 이 섹션은 구현 시 반드시 그대로 따라야 하는 정규 규칙이다. 모든 검증 로직은 여기에 근거한다.

### 2.1 구성물

- **보석 토큰 40개**: 5색 × 7개 + 골드(조커) 5개
  - 색상: 다이아몬드(흰색), 사파이어(파랑), 에메랄드(초록), 루비(빨강), 오닉스(검정)
  - 골드는 조커로, 구매 시 어떤 색이든 1개 대체 가능
- **개발 카드 90장**: Level 1 = 40장, Level 2 = 30장, Level 3 = 20장
- **귀족 타일 10장**: 각 3 명성점

### 2.2 카드 구조 (개발 카드 1장이 갖는 정보)

| 위치 | 의미 |
|---|---|
| 좌상단 숫자 | 명성 점수 (0~5) |
| 우상단 보석 | 이 카드가 제공하는 **영구 보너스 색** (구매 후 해당 색 토큰 1개로 영구 취급) |
| 하단 비용 | 구매에 필요한 색별 토큰 수 |

### 2.3 귀족 타일

- 각 귀족은 특정 **보너스 색 조합 요구치**를 가짐 (예: "초록3·빨강3·검정3" 또는 "흰4·파랑4" 형태).
- 요구치는 **토큰이 아니라 보유 카드 보너스**로 판정한다.
- 명성 3점. 게임 중 잃지 않음.

### 2.4 인원별 셋업 변수

| 인원 | 색별 토큰 수 | 골드 | 공개 귀족 수 |
|---|---|---|---|
| 2인 | 4개 | 5개 | 3장 |
| 3인 | 5개 | 5개 | 4장 |
| 4인 | 7개 | 5개 | 5장 |

- 개발 카드: 레벨별로 섞어 3개의 덱을 만들고, **각 레벨에서 4장씩 공개**(4×3 그리드).
- 귀족: (인원수 + 1)장 공개, 나머지는 게임에서 제외.
- 시작 플레이어는 무작위. 시계방향 진행.

### 2.5 턴 행동 — 매 턴 아래 4가지 중 **정확히 하나만** 선택

**(A) 서로 다른 색 보석 3개 가져오기**
- 서로 다른 3색에서 1개씩.
- 가져올 수 있는 색이 3색 미만이면, 가능한 만큼만 가져온다.

**(B) 같은 색 보석 2개 가져오기**
- 단, **해당 색 더미에 토큰이 4개 이상 남아 있을 때만** 가능 (가져간 뒤 최소 2개가 남아야 함).

**(C) 카드 1장 예약 + 골드 1개 받기**
- 테이블의 공개 카드를 예약하거나, 세 덱 중 한 덱의 맨 위 카드를 **블라인드로** 예약 가능.
- 예약 시 골드가 남아 있으면 1개 받는다. **골드가 없으면 골드 없이 예약만** 한다.
- 예약 카드는 **최대 3장**까지 보유 가능. 예약 카드는 버릴 수 없다.
- 예약 카드는 본인만 구매 가능(비공개).

**(D) 개발 카드 1장 구매**
- 테이블의 공개 카드 또는 **본인의 예약 카드**를 구매.
- 비용 지불 시 **보유 카드 보너스가 먼저 비용을 상쇄**하고, 부족분만 토큰으로 지불.
- **골드는 부족한 어떤 색이든 1개씩 대체** 가능.
- 지불한 토큰은 공급처로 반환. 골드도 반환.
- 구매한 카드는 앞면으로 본인 앞에 배치(보너스·점수 영구 획득).

### 2.6 턴 종료 시 규칙

1. **토큰 보유 한도 = 10개** (골드 포함). 턴 종료 시 10개를 초과하면 초과분을 반환(가져온 것 일부 또는 기존 보유분 중 선택해 반환).
   - 단, **카드 보너스는 토큰이 아니므로 한도에 포함되지 않는다.**
2. **귀족 방문 판정**: 턴 종료 시 보유 카드 보너스가 어떤 귀족의 요구치를 충족하면 그 귀족이 방문(명성 3점).
   - 동시에 여러 귀족 조건을 만족해도 **턴당 1명만** 선택해 받는다.
   - 귀족 방문은 행동을 소모하지 않음(자동 처리, 다중 충족 시 선택).

### 2.7 게임 종료 & 승리

- 어떤 플레이어가 **15 명성점 이상**에 도달하면 그 턴 이후 **현재 라운드를 끝까지 진행**(모든 플레이어가 동일 턴 수를 갖도록).
- 라운드 종료 후 **최고 명성점자 승리**.
- **동점 시**: **구매한 개발 카드 수가 더 적은** 플레이어 승리.

## 5. 데이터 모델 (TypeScript)

```ts
type GemColor = 'white' | 'blue' | 'green' | 'red' | 'black';
type TokenColor = GemColor | 'gold';
type CardLevel = 1 | 2 | 3;

// 토큰/비용은 색→개수 맵
type TokenPool = Record<TokenColor, number>; // gold 포함
type GemCost  = Record<GemColor, number>;    // gold 제외(비용엔 골드 없음)
type BonusMap = Record<GemColor, number>;    // 카드 보너스 누적

interface Card {
  id: string;
  level: CardLevel;
  prestige: number;       // 0~5
  bonus: GemColor;        // 영구 보너스 색
  cost: GemCost;          // 색별 요구 토큰
}

interface Noble {
  id: string;
  prestige: 3;
  requirement: BonusMap;  // 색별 요구 카드 보너스 수
}

interface Player {
  id: string;
  name: string;
  isAI: boolean;
  tokens: TokenPool;      // gold 포함, 합계 ≤ 10
  bonuses: BonusMap;      // 구매 카드 색별 누적
  reserved: Card[];       // 최대 3
  purchased: Card[];      // 구매 카드(보너스/점수 산출 원장)
  nobles: Noble[];
  prestige: number;       // 카드 + 귀족 합산
}

interface GameState {
  players: Player[];
  pool: TokenPool;                                   // 공급처 토큰
  board: Record<CardLevel, (Card | null)[]>;         // 각 레벨 공개 슬롯(최대 4)
  decks: Record<CardLevel, Card[]>;                  // 비공개 잔여 덱
  nobles: Noble[];                                   // 공개 귀족
  currentPlayerIndex: number;
  turnCount: number;
  phase: 'setup' | 'playing' | 'finalRound' | 'finished';
  finalRoundTriggeredBy: string | null;             // 15점 최초 도달자 id
  winnerId: string | null;
  log: GameEvent[];                                  // 행동 기록(리플레이/디버깅)
}
```

### 5.1 액션 타입

```ts
type Action =
  | { type: 'TAKE_THREE'; colors: GemColor[] }        // 길이 1~3, 서로 다름
  | { type: 'TAKE_TWO'; color: GemColor }
  | { type: 'RESERVE'; source: { from: 'board'; level: CardLevel; slot: number }
                              | { from: 'deck'; level: CardLevel } }
  | { type: 'PURCHASE'; source: { from: 'board'; level: CardLevel; slot: number }
                              | { from: 'reserved'; cardId: string };
      // 골드를 어디에 쓸지 자동 최적화하거나, 명시적 paymentPlan을 받음
      payment?: Record<TokenColor, number> }
  | { type: 'DISCARD_TOKENS'; tokens: Partial<TokenPool> }   // 10개 초과 시
  | { type: 'CHOOSE_NOBLE'; nobleId: string };               // 다중 충족 시
```

## 6. 게임 엔진 — 검증 & 처리 로직

> 모든 행동은 `validate(state, action)` 통과 후 `reduce(state, action)`로 적용. UI는 검증 실패하는 버튼을 미리 비활성화한다.

### 6.1 TAKE_THREE (서로 다른 3색)
- 검증: `colors`는 서로 다른 1~3개, 각 색 `pool[color] >= 1`.
- 처리: 각 색 1개씩 플레이어로 이동.
- 종료 후: 토큰 한도 검사 → 귀족 판정.

### 6.2 TAKE_TWO (같은 색 2개)
- 검증: `pool[color] >= 4` (핵심 규칙).
- 처리: 해당 색 2개 이동.

### 6.3 RESERVE
- 검증: `player.reserved.length < 3`. board 출처면 슬롯에 카드 존재, deck 출처면 덱 비어있지 않음.
- 처리: 카드를 `reserved`로 이동, board 슬롯이면 덱에서 보충(있으면). `pool.gold >= 1`이면 골드 1개 지급(없으면 미지급).

### 6.4 PURCHASE
- **지불 가능 판정**:
  ```
  각 색별 부족분 = max(0, cost[color] - bonuses[color] - tokens[color])
  필요한 골드 = Σ 부족분
  구매 가능 ⟺ 필요한 골드 ≤ tokens.gold
  ```
- 처리: 색별로 `max(0, cost - bonus)`만큼을 우선 해당 색 토큰으로, 모자라면 골드로 지불 → 지불 토큰은 pool로 반환. 카드를 `purchased`로 이동, `bonuses[card.bonus]++`, `prestige += card.prestige`. board 출처면 슬롯 보충.
- **골드 사용 최적화**: 기본은 "해당 색 토큰을 최대한 쓰고 부족분만 골드"로 충분. UI에서 수동 지불 플랜을 허용할지는 선택(MVP는 자동).

### 6.5 토큰 한도 (턴 종료)
- `Σ tokens > 10`이면 `DISCARD_TOKENS` 요구. 사람: UI로 선택 반환. AI: 휴리스틱(목표 카드에 덜 필요한 색부터 반환).

### 6.6 귀족 판정 (턴 종료)
- `eligible = nobles.filter(n => 모든 색 bonuses[color] >= n.requirement[color])`
- `eligible.length === 1` → 자동 획득.
- `eligible.length > 1` → `CHOOSE_NOBLE` 요구(사람 선택 / AI 휴리스틱).
- 획득 시 해당 귀족을 공개 목록에서 제거, `prestige += 3`.

### 6.7 종료 판정 (턴 종료 후)
- 누군가 `prestige >= 15`이고 아직 `finalRoundTriggeredBy === null`이면 설정하고 `phase = 'finalRound'`.
- finalRound 중 시작 플레이어 차례가 다시 돌아와 한 바퀴가 완결되면 `phase = 'finished'`, 승자 산정(2.7).

## 7. AI 상대 설계 (휴리스틱)

MVP는 규칙 기반 가중치 AI. 난이도는 가중치/탐색 깊이로 조절.

**의사결정 우선순위(예시 정책)**
1. 이번 턴에 **즉시 구매 가능한 카드** 중 점수 효율이 높은 것 구매(특히 귀족 진척 + 명성점).
2. 구매 불가하면, **가장 가까운 목표 카드**(필요 토큰 수 최소)를 정해 그 색 토큰 확보(가능하면 TAKE_TWO, 아니면 TAKE_THREE).
3. 상대가 곧 가져갈 고가치 카드는 **선예약**으로 견제 + 골드 확보.
4. 토큰 10개 임박 시 구매 우선.

**평가 함수 요소**: 명성점, 귀족까지 남은 보너스 수, 보너스 색 다양성, 골드 보유, 상대 위협(15점 근접도).

- 난이도 3단계: Easy(탐욕적, 1수), Normal(가중 평가, 1수), Hard(2~3수 시뮬레이션 + 견제 고려).
- 엔진이 순수 함수이므로 AI는 `applyAction`을 가상 적용해 후보 수를 평가하면 됨.

## 8. UI/UX 화면 구성

**메인 게임 보드(1화면 집약)**
- 상단: 공개 귀족 타일(요구 보너스 아이콘 표시).
- 중앙: 3×4 개발 카드 그리드(레벨3 위→레벨1 아래). 각 카드에 점수/보너스색/비용, **구매가능 여부 하이라이트**.
- 좌/우: 공급처 토큰 더미(색별 잔량 표시, TAKE_TWO 비활성 조건 시각화).
- 하단: 현재 플레이어 패널(보유 토큰, 색별 보너스 카운트, 예약 카드, 명성점).
- 사이드: 다른 플레이어 요약(점수·카드수·예약수, 예약 내용은 비공개).

**핵심 인터랙션**
- 불가능한 행동은 버튼/카드 비활성 + 사유 툴팁("이 색은 4개 미만이라 2개를 가져올 수 없습니다").
- 토큰 선택은 클릭 토글, "확정" 전 미리보기.
- 구매 시 골드 자동 충당 표시, 잔여 골드/토큰 변화 프리뷰.
- 10개 초과 시 반환 모달, 다중 귀족 시 선택 모달.
- 행동 로그 패널(리플레이/학습용).

**접근성/표현**: 색맹 대응(색+아이콘+라벨 병기), 애니메이션은 토큰 이동/카드 획득 정도로 절제.

## 10. 시드 데이터 (90 카드 + 10 귀족)

> **중요**: 90장 카드와 10장 귀족의 정확한 비용/보너스/점수 값은 게임 고유 데이터다. 본 PRD는 **스키마와 예시**만 제공하며, 구현 시 검증된 공개 데이터셋(BoardGameGeek 자료 또는 오픈소스 Splendor 구현 repo의 JSON)을 `/data/cards.json`, `/data/nobles.json`으로 시드한 뒤, **레벨별 장수(40/30/20)와 귀족 10장**이 맞는지 자동 검증 스크립트로 확인할 것.

```jsonc
// cards.json (예시 항목)
{ "id": "L1-001", "level": 1, "prestige": 0, "bonus": "black",
  "cost": { "white": 1, "blue": 1, "green": 1, "red": 1, "black": 0 } }

// nobles.json (예시 항목)
{ "id": "N-01", "prestige": 3,
  "requirement": { "white": 0, "blue": 0, "green": 4, "red": 4, "black": 0 } }
```

**시드 검증 체크리스트**
- [ ] Level1 = 40, Level2 = 30, Level3 = 20 (합 90)
- [ ] 귀족 = 10
- [ ] 모든 카드 `bonus`는 5색 중 하나, `prestige`는 0~5
- [ ] 모든 귀족 `prestige === 3`

## 11. 엣지 케이스 체크리스트 (테스트 필수)

- [ ] TAKE_TWO: 해당 색이 정확히 3개일 때 **불가**, 4개일 때 가능.
- [ ] TAKE_THREE: 잔여 색이 2종뿐이면 2개만 가져가기 허용.
- [ ] RESERVE: 골드가 0개여도 예약은 가능(골드만 미지급).
- [ ] RESERVE: 예약 3장 보유 시 추가 예약 차단.
- [ ] RESERVE: 덱이 비었으면 해당 레벨 블라인드 예약 불가.
- [ ] PURCHASE: 보너스만으로 비용 전액 충당되는 0코스트 구매(토큰 0 지불) 정상 처리.
- [ ] PURCHASE: 골드로 부족분 정확히 충당, 골드 초과 보유분은 안 씀.
- [ ] 토큰 한도: 11개 이상 시 반환 강제, 카드 보너스는 한도에서 제외.
- [ ] 귀족 다중 충족: 1명만 선택, 나머지는 남음.
- [ ] 종료: 15점 도달해도 **현재 라운드 끝까지** 진행, 동점 시 카드 수 적은 쪽 승.
- [ ] 보충: 카드 구매/예약 후 해당 레벨 덱이 비었으면 슬롯은 빈 채로 유지.
